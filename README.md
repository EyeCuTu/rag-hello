# rag-hello

Учебный RAG-проект (Retrieval-Augmented Generation): задаёшь вопрос на естественном языке — система находит релевантные фрагменты в PDF-документе и формирует ответ через Claude, опираясь только на контекст из документа.

Первая практическая работа в треке прикладного LLM. Цель — пройти полный пайплайн RAG руками и понять каждый компонент.

## Стек

- **Python 3.12**
- **LangChain** — оркестрация пайплайна
- **Chroma** — локальная векторная база
- **HuggingFace Embeddings** (`BAAI/bge-m3`) — мультиязычные эмбеддинги, бесплатные, локальные. Подбор модели обоснован в разделе «Эксперимент».
- **Anthropic Claude** (`claude-haiku-4-5`) — генерация ответов

## Архитектура

Проект состоит из трёх скриптов:

- **`index.py`** — индексация. Загружает PDF из `data/`, режет на чанки (1000 символов с overlap 200), считает эмбеддинги, сохраняет в Chroma. Идемпотентен: при повторном запуске пересоздаёт базу с нуля.
- **`query.py`** — поиск и ответ. Открывает существующую базу, для каждого вопроса находит top-5 релевантных чанков, склеивает в контекст, отправляет Claude с защитным промптом, печатает ответ.
- **`eval.py`** — оценка качества retrieval. Прогоняет вопросы из `eval_questions.json` и считает метрики `hit_rate@k` и `MRR@k`. См. раздел «Эксперимент».

Разделение «дорогая индексация — дешёвые запросы» — стандартная практика для production RAG.

```bash
            langchain-huggingface chromadb pypdf python-dotenv sentence-transformers

# Положить любой PDF в data/integrals.pdf (или поправить путь в index.py)

# (опционально) Замерить hit_rate@k и MRR@k на evaluation set
python eval.py
```

## Тестовые результаты

Проиндексирована методичка РГУ нефти и газа им. Губкина «Кратные и криволинейные интегралы» (45 страниц, русский язык, ~90 чанков).

Прогон тестовых вопросов показывает два классических сценария поведения RAG:

**1. Полный ответ.** «Как вычислить площадь плоской фигуры с помощью двойного интеграла?» — ретривер находит соответствующий раздел методички, Claude собирает структурированный ответ с обеими формулами (декартовы и полярные координаты) и ссылается на конкретные примеры из текста.

**2. Отказ.** «Как Лейбниц придумал интегральное исчисление?» — в методичке нет исторической справки. Claude не использует свои собственные знания о Лейбнице, строго опирается на контекст и отвечает «в предоставленных материалах нет ответа».

Это и есть цель защитного промпта: предсказуемое поведение в обоих сценариях.

## Эксперимент: качество retrieval

Для измеримой оценки качества поиска собран evaluation set из 20 вопросов с указанием правильных страниц PDF (`eval_questions.json`). Используются две метрики:

- **`hit_rate@k`** — доля запросов, у которых хотя бы одна из правильных страниц попадает в top-k результатов. Бинарная: «нашли / не нашли».
- **`MRR@k`** (Mean Reciprocal Rank) — среднее значение `1/позиция первого правильного ответа`. Чувствительна к ранжированию: ответ на первом месте даёт 1.0, на втором — 0.5, на третьем — 0.33. Показывает, насколько хорошо ретривер не просто *находит*, а *приоритизирует* правильные чанки.

Скрипт `eval.py` автоматически прогоняет evaluation set и считает обе метрики. Запуск: `python eval.py`.

### Шаг 1. Базовая модель vs `bge-m3` (5 вопросов, k=5)

Стартовый замер на минимальном наборе вопросов, чтобы оправдать смену embedding-модели:

| Модель | Размер вектора | hit_rate@5 |
|---|---|---|
| `paraphrase-multilingual-MiniLM-L12-v2` | 384 | 80% (4/5) |
| `BAAI/bge-m3` | 1024 | 100% (5/5) |

Главная победа `bge-m3` — на абстрактных запросах вида «что такое X». Маленькая multilingual-модель интерпретировала такие запросы как поиск ключевого словосочетания и возвращала примеры использования вместо определения. `bge-m3` правильно различает «определение X» и «применение X».

Цена: модель `bge-m3` весит ~2.3 ГБ против ~470 МБ у MiniLM, индексация на CPU заметно дольше. На GPU (RTX 4060) индексация 90 чанков занимает секунды.

### Шаг 2. Расширенный evaluation set и MRR (20 вопросов, k=3)

После смены модели evaluation set расширен до 20 вопросов разных типов (определения, формулы, процедуры, теоремы — по всем трём разделам методички), `top_k` снижен до 3 для более жёсткой оценки. Добавлена метрика MRR.

| Модель | hit_rate@3 | MRR@3 |
|---|---|---|
| `BAAI/bge-m3` | 100% (20/20) | 0.775 |

`hit_rate@3 = 100%` означает, что для каждого из 20 вопросов правильная страница есть в top-3. Но `MRR = 0.775` показывает, что **на первом месте** правильный ответ оказывается не всегда — средняя позиция между первым (1.0) и вторым (0.5).

Из 20 вопросов: 11 имеют RR = 1.0 (правильная страница на 1-м месте), 5 — RR = 0.5 (на 2-м месте), 4 — RR = 0.333 (на 3-м месте).

Слабые места — вопросы, где термин из теоретического раздела также встречается в разделах задач и упражнений. Ретривер на этих вопросах ставит на первое место чанки с задачами, а определение — на второе или третье. Эту проблему планирует решить hybrid search — следующий эксперимент.

## Что работает

- Полный пайплайн RAG (load → split → embed → store → retrieve → augment → generate).
- Защита от галлюцинаций через промпт.
- Идемпотентная индексация без накопления дубликатов в Chroma.
- Измеримая оценка качества retrieval через hit_rate и MRR на собственном evaluation set.

## Известные ограничения

- **Извлечение PDF теряет структуру формул**: матан-выражения попадают в чанки сырым текстовым дампом. Это ухудшает качество эмбеддингов на чанках с большим количеством формул. Решение — использовать специализированные парсеры PDF (например, `unstructured`).
- **Нет переписывания запроса** перед поиском. Реальные RAG-системы прогоняют пользовательский вопрос через LLM, чтобы привести его к языку документа, до того как идти в векторную базу.
- **Ретривер путает теоретические разделы с разделами задач** при общих терминах (см. результаты эксперимента, шаг 2). Цель hybrid search — снизить эту путаницу.

## Что дальше

- Hybrid search через Chroma + BM25 — с целью улучшения MRR за счёт точного поиска по терминам.
- Reranking найденных чанков через cross-encoder.
- Веб-интерфейс (FastAPI + простой фронт).
- Чтение PDF через специализированный парсер с сохранением структуры формул.
- Дальнейшее расширение evaluation set, особенно вопросами-ловушками (на которые ответа в документе нет).
